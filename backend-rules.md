# AI Rules — Node.js TypeScript Backend

> **Stack:** Modular · Clean/Hexagonal · Hybrid CQRS · Manual DI · Hono · Zod · Drizzle (query builder) · OpenTelemetry · Pino · Cache · Unit of Work · Outbox · Inbox · Biome

---

## 1. Project Philosophy

- **Modular first** — every feature is a self-contained module under `src/modules/<n>/`
- **Clean / Hexagonal** — domain is framework-agnostic; infrastructure adapts via ports & adapters
- **Hybrid CQRS** — apply Command/Query split only when complexity warrants it; use plain Controller → Service for simple requests
- **Manual DI** — no DI container library; wire all dependencies explicitly in `src/infra/container.ts`
- **Drizzle as query builder only** — schema generated via `drizzle-kit pull`; Drizzle never owns the schema
- **Migrations via raw SQL** — every migration has a paired `.up.sql` / `.down.sql` with a custom runner
- **Patterns are opt-in** — Cache, Unit of Work, Outbox, Inbox applied per-module based on need

## 3. TypeScript Conventions

- `strict: true` in `tsconfig.json` — enables `strictNullChecks`, `noImplicitAny`, etc.
- `const` by default; `let` only when reassignment is needed; never `var`
- `interface` for object shapes that may be extended; `type` for unions, intersections, mapped types
- `unknown` over `any` — force type narrowing before use; never use `any`
- Use discriminated unions for state variants — not boolean flags
- `readonly` for arrays and properties that must not be mutated
- Explicit return types on all exported functions — never rely on inference for public APIs
- `satisfies` operator for type validation without widening: `const config = { ... } satisfies Config`
- If a type is used in more than 2 files → move to `shared/types/`; otherwise co-locate
- Use `WeakMap` / `WeakSet` for caches keyed on object references

**Fallible operations — prefer Result types over throwing:**

```typescript
type Result<T, E = AppError> =
  | { ok: true;  data: T }
  | { ok: false; error: E };

async function findUser(id: string): Promise<Result<User>> {
  const row = await db.select()...
  if (!row) return { ok: false, error: new NotFoundError(`User ${id}`) };
  return { ok: true, data: UserMapper.toDomain(row) };
}
```

**JSDoc on all public APIs:**

```typescript
/**
 * Places a new order for the given customer.
 *
 * @param dto - Validated order creation payload.
 * @returns The new order ID.
 * @throws {DomainError} When the item list is empty.
 */
async execute(dto: PlaceOrderDto): Promise<string> { ... }
```

## 5. Hybrid CQRS — Decision Guide

```
Request comes in
      │
      ├─ Has domain logic / events / Outbox?  →  CQRS (Command / Query handler)
      │
      └─ Simple data operation?               →  Controller → Service → Repository
```

**Use CQRS when any of these are true:**
- Write and read models diverge (aggregates vs. flat projections)
- The use-case publishes domain events or needs an Outbox
- The operation needs an audit trail or complex invariant checks

**Use plain Service when:**
- Simple CRUD with no domain rules
- Read-only endpoint with no side effects
- Internal utility (lookup, config fetch, ping)

## 6. Manual Dependency Injection

No `tsyringe`, `awilix`, or `reflect-metadata`. Constructor injection only — typed via interfaces.

```typescript
// infra/container.ts
import { db }    from './db';
import { redis } from './redis';

const logger     = createLogger();
const uow        = new DrizzleUnitOfWork(db);
const cache      = new RedisCache(redis);

// Order module
const orderRepo  = new OrderRepository(db);
const outbox     = new DrizzleOutbox(db);
const placeOrder = new PlaceOrderHandler(orderRepo, uow, outbox, logger);
const getOrder   = new GetOrderHandler(new OrderReadModel(db), cache, logger);
const orderCtrl  = new OrderController(placeOrder, getOrder, logger);
const orderRouter = createOrderRouter(orderCtrl);

// Health
const health = new HealthCheckRegistry();
health.register(new DbHealthIndicator(db));
health.register(new RedisHealthIndicator(redis));

export { orderRouter, health };
```

**Rules:**
- Constructor params typed via ports (interfaces), never concrete classes
- Wire once at startup — never `new Dependency()` inside business logic
- Each module's `index.ts` exports its wired instances; `container.ts` assembles the app

## 7. Layer Rules

### 7.1 Domain Layer (`domain/`)

Pure TypeScript — zero external imports. Entities expose behaviour. Ports are `interface` only.

```typescript
// modules/order/domain/entities/order.entity.ts
export class Order {
  private readonly _events: DomainEvent[] = [];

  static create(id: string, items: readonly OrderItem[]): Order { ... }

  confirm(): void {
    if (this.status !== 'pending') throw new DomainError('Cannot confirm a non-pending order');
    this.status = 'confirmed';
    this._events.push(new OrderConfirmedEvent(this.id));
  }

  pullEvents(): DomainEvent[] { return this._events.splice(0); }
}
```

### 7.2 Application Layer — CQRS path

```typescript
// application/commands/place-order.handler.ts
export class PlaceOrderHandler {
  constructor(
    private readonly repo: IOrderRepository,
    private readonly uow: IUnitOfWork,
    private readonly outbox: IOutbox,
    private readonly logger: ILogger,
  ) {}

  /** @throws {DomainError} When items list is empty. */
  async execute(dto: PlaceOrderDto): Promise<string> {
    return withSpan('PlaceOrderHandler.execute', async (span) => {
      const order = Order.create(dto.id, dto.items);

      await this.uow.run(async (tx) => {
        await this.repo.save(order, tx);
        await this.outbox.publish(order.pullEvents(), tx); // atomic
      });

      span.setAttribute('order.id', order.id);
      return order.id;
    });
  }
}
```

### 7.3 Application Layer — Simple Service path

```typescript
// application/services/product-lookup.service.ts
export class ProductLookupService {
  constructor(
    private readonly repo: IProductRepository,
    private readonly cache: ICache,
  ) {}

  async findById(id: string): Promise<ProductDto> {
    return getCached(this.cache, `product:${id}`, async () => {
      const product = await this.repo.findById(id);
      if (!product) throw new NotFoundError(`Product ${id} not found`);
      return ProductMapper.toDto(product);
    }, 300);
  }
}
```

### 7.4 Infrastructure Layer (`infrastructure/`)

- Drizzle used as a **query builder** inside `persistence/` and `read-model/` only
- Map Drizzle rows → domain entities or DTOs — never expose raw rows to callers
- Use `$inferSelect` / `$inferInsert` for insert/select types derived from the pulled schema

```typescript
// infrastructure/persistence/order.repository.ts
export class OrderRepository implements IOrderRepository {
  constructor(private readonly db: DrizzleClient) {}

  async save(order: Order, tx?: Transaction): Promise<void> {
    const client = tx ?? this.db;
    await client
      .insert(orderTable)
      .values(OrderMapper.toPersistence(order))
      .onConflictDoUpdate({ target: orderTable.id, set: { status: order.status } });
  }

  async findById(id: string): Promise<Order | null> {
    const row = await this.db
      .select().from(orderTable)
      .where(eq(orderTable.id, id))
      .limit(1)
      .then((r) => r[0] ?? null);
    return row ? OrderMapper.toDomain(row) : null;
  }
}
```

## 9. Zod — Validation

Validate at system boundaries: API input, env vars, config files. Never inside domain.

```typescript
// shared/types/env.ts
const EnvSchema = z.object({

  DATABASE_URL:                z.string().url(),
```


```typescript
// database/schema/orders.ts  ← GENERATED — DO NOT EDIT
export const orderTable = pgTable('orders', {
```

## 11. Database Migrations — `.up.sql` / `.down.sql`

### Layout

```
database/migrations/
├── 20240101000000_init/
│   ├── up.sql
│   └── down.sql
├── 20240202120000_add_outbox/
│   ├── up.sql
│   └── down.sql
└── 20240310090000_add_inbox/
    ├── up.sql
    └── down.sql
```

Folder format: `YYYYMMDDHHMMSS_<description>`

### Example

```sql
-- 20240202120000_add_outbox/up.sql
CREATE TABLE outbox (
  id           UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  aggregate_id TEXT        NOT NULL,
  event_type   TEXT        NOT NULL,
  payload      JSONB       NOT NULL,
  published    BOOLEAN     NOT NULL DEFAULT FALSE,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_outbox_unpublished ON outbox (published, created_at) WHERE published = FALSE;
```

```sql
-- 20240202120000_add_outbox/down.sql
DROP TABLE IF EXISTS outbox;
```

### Custom Runner

```typescript
// database/migrate.ts
export async function runMigrations(direction: 'up' | 'down' = 'up'): Promise<void> {
  await db.execute(sql`
    CREATE TABLE IF NOT EXISTS _migrations (
      version    TEXT PRIMARY KEY,
      applied_at TIMESTAMPTZ DEFAULT NOW()
    )
  `);

  const applied = new Set(
    (await db.select({ version: _migrations.version }).from(_migrations)).map((r) => r.version),
  );

  const dirs = (await readdir(MIGRATIONS_DIR)).sort();
  const targets = direction === 'up'
    ? dirs.filter((d) => !applied.has(d))
    : dirs.filter((d) =>  applied.has(d)).reverse();

  for (const dir of targets) {
    const sqlText = await readFile(join(MIGRATIONS_DIR, dir, `${direction}.sql`), 'utf-8');
    await db.transaction(async (tx) => {
      await tx.execute(sql.raw(sqlText));
      direction === 'up'
        ? await tx.insert(_migrations).values({ version: dir })
        : await tx.delete(_migrations).where(eq(_migrations.version, dir));
    });
    console.info(`[migrate:${direction}] ${dir}`);
  }
}
```

### Scripts

```json
{
  "scripts": {
    "db:pull":     "drizzle-kit pull",
    "db:migrate":  "tsx src/database/migrate.ts up",
    "db:rollback": "tsx src/database/migrate.ts down",
    "db:studio":   "drizzle-kit studio"
  }
}
```

**Rules:** never edit a committed migration · always pair up+down · test rollback locally · run `db:pull` after applying · `db:migrate` runs as init step before app start in production.

## 12. Unit of Work

Apply when multiple writes must be **atomic**.

```typescript
// shared/unit-of-work/unit-of-work.interface.ts
export interface IUnitOfWork {
  run<T>(work: (tx: Transaction) => Promise<T>): Promise<T>;
}

// shared/unit-of-work/drizzle-unit-of-work.ts
export class DrizzleUnitOfWork implements IUnitOfWork {
  constructor(private readonly db: DrizzleClient) {}
  run<T>(work: (tx: Transaction) => Promise<T>): Promise<T> {
    return this.db.transaction(work);
  }
}
```

| Scenario | UoW? |
|----------|------|
| Save aggregate + publish Outbox in same tx | ✅ Yes |
| Multiple aggregates updated together | ✅ Yes |
| Read-only query | ❌ No |
| Single-table insert, no side effects | ❌ No |

## 13. Cache Pattern

Apply for read-heavy data that is expensive to recompute.

```typescript
// shared/cache/cache.interface.ts
export interface ICache {
  get<T>(key: string): Promise<T | null>;
  set<T>(key: string, value: T, opts?: { ttlSeconds?: number }): Promise<void>;
  del(key: string): Promise<void>;
  invalidateByPrefix(prefix: string): Promise<void>;
}

// Cache-aside helper
export async function getCached<T>(
  cache: ICache,
  key: string,
  fetch: () => Promise<T>,
  ttlSeconds = 60,
): Promise<T> {
  const hit = await cache.get<T>(key);
  if (hit !== null) return hit;
  const value = await fetch();
  await cache.set(key, value, { ttlSeconds });
  return value;
}
```

**Rules:**
- Key format: `<module>:<entity>:<id>` — e.g. `order:product:abc123`
- Invalidate on write, not on read
- Always set a TTL — never cache without expiry
- Do not cache transactional or strongly-consistent data

| Scenario | Cache? |
|----------|--------|
| Read-heavy, rarely mutated (catalogue, config) | ✅ Yes |
| User session / token data | ✅ Yes (short TTL) |
| Write path / transactional data | ❌ No |
| Strong consistency required | ❌ No |

## 14. Outbox Pattern

Apply when a domain event **must** reach a message broker (at-least-once delivery).

```typescript
export interface IOutbox {
  publish(events: DomainEvent[], tx: Transaction): Promise<void>;
}

export class DrizzleOutbox implements IOutbox {
  async publish(events: DomainEvent[], tx: Transaction): Promise<void> {
    if (!events.length) return;
    await tx.insert(outboxTable).values(
      events.map((e) => ({ aggregateId: e.aggregateId, eventType: e.type, payload: e })),
    );
  }
}
```

**Dispatcher** (background worker — polls unpublished rows):

```typescript
export class OutboxDispatcher {
  async dispatch(): Promise<void> {
    const rows = await this.db.select().from(outboxTable)
      .where(eq(outboxTable.published, false))
      .orderBy(asc(outboxTable.createdAt)).limit(50);

    for (const row of rows) {
      await this.db.transaction(async (tx) => {
        await this.broker.publish(row.eventType, row.payload);
        await tx.update(outboxTable).set({ published: true }).where(eq(outboxTable.id, row.id));
      });
    }
  }
}
```

| Scenario | Outbox? |
|----------|---------|
| Event must reach broker after domain write | ✅ Yes |
| Fire-and-forget (loss acceptable) | ❌ No |
| In-process event bus only | ❌ No |

```typescript
export class OrderConfirmedConsumer {
```

## 16. Error Handling

**Error taxonomy:**

```typescript
// shared/errors/app-error.ts
export class AppError extends Error {
  constructor(
    readonly message: string,
    readonly code: string,
    readonly httpStatus: number = 500,
    readonly context?: Record<string, unknown>,
  ) { super(message); this.name = this.constructor.name; }
}

export class DomainError     extends AppError { constructor(msg: string) { super(msg, 'DOMAIN_ERROR',     422); } }
export class NotFoundError   extends AppError { constructor(msg: string) { super(msg, 'NOT_FOUND',        404); } }
export class ConflictError   extends AppError { constructor(msg: string) { super(msg, 'CONFLICT',         409); } }
export class ValidationError extends AppError { constructor(msg: string) { super(msg, 'VALIDATION_ERROR', 400); } }
export class UnauthorizedError extends AppError { constructor(msg: string) { super(msg, 'UNAUTHORIZED',   401); } }
```

**Rules:**
- `AppError` subclasses for **expected** business failures (validation, not-found, conflict)
- Thrown exceptions for **infrastructure failures** (network, I/O, OOM)
- Wrap third-party exceptions at module boundaries into domain-specific error types
- Always attach context: operation name, input IDs, correlation/trace ID
- Never swallow exceptions silently — log or propagate with context
- Use `Result<T, E>` return type for recoverable operations where callers must handle both paths
- Only retry **transient** errors (network timeout, 503) — never retry 400/404; use exponential backoff with a max retry count
- Unhandled promise rejections must crash the process in production

**Structured error response:**

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Items list cannot be empty",
    "details": [{ "field": "items", "message": "Required" }]
  }
}
```

```typescript
// shared/telemetry/with-span.ts
export async function withSpan<T>(
  name: string,
  fn: (span: Span) => Promise<T>,
): Promise<T> {
  return tracer.startActiveSpan(name, async (span) => {
    try {
      const result = await fn(span);
      span.setStatus({ code: SpanStatusCode.OK });
      return result;
    } catch (err) {
      span.recordException(err as Error);
      span.setStatus({ code: SpanStatusCode.ERROR, message: String(err) });
      throw err;
    } finally {
      span.end();
    }
  });
}
```

## 20. Performance

- `for` / `for...of` over `forEach` / `map` in hot paths
- Avoid object allocations inside tight loops
- `Map` / `Set` for frequent insertions, deletions, and keyed lookups
- `structuredClone()` for deep cloning — never `JSON.parse(JSON.stringify())`
- Never use synchronous I/O methods — always `async/await`
- `node:worker_threads` for CPU-intensive tasks to avoid blocking the event loop
- Typed arrays (`Uint8Array`, `Float64Array`) for large numerical or binary data streams
- Debounce / throttle high-frequency events
- Avoid wildcard imports (`import * as _`) — import only what is needed
- Use prepared statements (`.prepare()`) for hot Drizzle queries

## 21. Testing

**Unit tests** — for domain entities, value objects, application handlers, and pure utility functions:

```typescript
// Arrange-Act-Assert, one behaviour per test
describe('Order.confirm()', () => {
  it('should emit OrderConfirmedEvent when status is pending', () => {
    const order = Order.create('o1', [{ productId: 'p1', qty: 2 }]);
    order.confirm();
    expect(order.pullEvents()).toContainEqual(expect.objectContaining({ type: 'order.confirmed' }));
  });

  it('should throw DomainError when status is not pending', () => {
    const order = Order.create('o1', [{ productId: 'p1', qty: 2 }]);
    order.confirm();
    expect(() => order.confirm()).toThrow(DomainError);
  });
});
```

**Rules:**
- Test one behaviour per test case — fast, isolated, deterministic
- Mock all external dependencies in unit tests (repos, cache, broker)
- Name tests: `should <expected behaviour> when <condition>`
- Use table-driven / parameterized tests for many input/output combinations
- Mirror source structure: `src/modules/order/domain/entities/order.entity.ts` → `order.entity.test.ts`
- 80%+ coverage on business logic; don't chase 100% — test behaviour, not implementation
- Write a failing test first when reproducing a bug, then fix
- No test should depend on another test's state or execution order

**Integration tests** — infrastructure layer (repositories, outbox, inbox) against a real DB (test containers).

**E2E tests** — critical HTTP flows via Hono test client:

```typescript
import { testClient } from 'hono/testing';

const client = testClient(app);
const res = await client.api.orders.$post({ json: { id: 'o1', items: [...] } });
expect(res.status).toBe(201);
```

## 23. API Design

- Plural nouns for resources; nested routes for relationships; no verbs in URLs
- Version from day one: `/v1/` URL prefix
- Consistent error format: `{ error: { code, message, details? } }`
- Cursor-based pagination preferred over offset; include `nextCursor` in response
- `?sort=-created_at&fields=id,name&filter=status:active` for list endpoints
- ETags / `Last-Modified` headers for caching; respond `304` when unchanged
- Rate limiting headers: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`
- Webhooks with HMAC-SHA256 signatures for payload verification
- Bulk endpoints for operations users would otherwise call in a loop
- Audit log all write operations: who · what · when · from which IP

## 26. Pattern Decision Matrix

| Situation | Pattern |
|-----------|---------|
| Complex domain logic + domain events | CQRS — Command handler |
| Read-only projection / reporting query | CQRS — Query handler |
| Simple CRUD, no domain rules | Plain Controller → Service |
| Multiple writes must be atomic | Unit of Work |
| Event must reach broker reliably | Outbox |
| Broker consumer must be idempotent | Inbox |
| Read-heavy, slow or expensive data | Cache (cache-aside) |
| Real-time / strong consistency | No cache |
| Recoverable business failure | Result type |
| Infrastructure failure | Throw exception |

## 27. Do / Don't Quick Reference

### ✅ Do

- Wire all dependencies manually in `infra/container.ts`
- Use `interface` for ports; `type` for unions and mapped types
- Use Drizzle only as a query builder — schema from `db:pull`
- Always pair `up.sql` + `down.sql`; test rollback locally
- Re-run `db:pull` after every migration
- Use `withSpan()` in every handler/service that does I/O
- Validate with Zod at HTTP boundary and env vars — nowhere else
- Apply Unit of Work when aggregate save + outbox must be atomic
- Invalidate cache on write; always set a TTL
- Use `node:` prefix for built-ins; `crypto.randomUUID()`; `structuredClone()`
- Run `biome ci .` in CI; `biome check --write .` locally

### ❌ Don't

- Import Hono / Drizzle / Redis / OTel types into `domain/` or `application/`
- Use `drizzle-kit generate` or `drizzle-kit push`
- Edit generated schema files in `database/schema/`
- Edit a migration file after it has been applied
- Instantiate dependencies inside business logic
- Use `any` — use `unknown` + type guards
- Use `var` or `.then().catch()` chains
- Use `dotenv`, `uuid`, `axios`, `express`, or DI container libraries
- Cache transactional or strongly-consistent data
- Skip the Inbox guard on broker consumers
- Swallow exceptions silently
